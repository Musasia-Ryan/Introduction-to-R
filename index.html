Reading Data into R

Before you can analyse data, you must load it into R. Two tidyverse packages handle virtually every file format you will encounter in practice.

Key Packages

  • readr reads rectangular text files.
  • readxl reads Excel workbooks.
  • Both produce a tibble — R’s modern, opinionated data frame.
Package Function Use when…
readr read_csv() Your file is .csv — the universal format
readr read_tsv() Columns are separated by tabs
readxl read_excel() Your file is .xlsx or .xls 
library(tidyverse)   
library(readxl)

readxl is not part of the tidyverse core. Always call library(readxl) explicitly.

Reading a CSV File

  • read_csv() is the workhorse for flat files. It reads faster than base R’s read.csv(), it never silently converts strings to factors, and prints a column-type summary on load so you always know what was parsed.
# reading the data 
# this should work when the file is in your working directory
ideal <- read_csv("ideal.csv")

Reading an Excel File

Excel workbooks carry hidden complexity including multiple sheets, merged header rows, dates stored as serial numbers.

  • read_excel() exposes each of these as an explicit argument, making your assumptions visible and reproducible.
# this reads the first sheet by default
ideal <- read_excel("ideal.xlsx")

# Precise reading, when sheets have merged titles or extra header rows
ideal <- read_excel(
  "ideal.xlsx", # excel file
  sheet = "Calf Data", # specified sheet name
  skip = 2, # skip merged title rows before the header
  col_names = TRUE # use the first row as column names,
)

The Pipe Operator & filter()

The pipe is the single most transformative idea in tidyverse programming. It turns nested, inside-out function calls into a left-to-right sentence.

The Pipe Operator |>

  • Without the pipe, reading R code means parsing from the innermost call outward — the opposite of how you reason about a problem.
  • The pipe lets you write steps in the order they happen.

Without the pipe

# Read from the inside out — unnatural
select(
  filter(ideal, ReasonsLoss1 == "survived"),
  CalfID, sublocation, Weight
)

With the pipe

# Read left to right — matches how you think
ideal |>
  dplyr::filter(ReasonsLoss1 == "survived") |>
  dplyr::select(CalfID, sublocation, Weight)

Read |> as “and then” — take the data, and then filter, and then select. Keyboard shortcut: Ctrl + Shift + M in RStudio.

%>% vs |> — Which Pipe?

%>% (base R) \|> (magrittr)
Requires a package? Yes — loaded with tidyverse No — built into R ≥ 4.1
Speed Slightly slower Marginally faster
Where you’ll see it Most existing tidyverse tutorials Modern R code, including this session 
# These two lines produce identical output
ideal %>% filter(ReasonsLoss1 == "survived")
ideal |>  filter(ReasonsLoss1 == "survived")

rename()

  • You have now seen the friction that raw source names create — backticks for Theileria.spp., uppercase everywhere, cryptic codes like ReasonsLoss1. rename() removes that friction permanently. Do it once, and every line that follows is cleaner.
# syntax: rename(new_name = old_name)
ideal |>
  rename(
    sex       = CalfSex,
    theileria = `Theileria.spp.`,
    outcome   = ReasonsLoss1

  select(CalfID, sex, theileria, outcome) |>
  head(2)

mutate()

  • Raw data rarely arrives in the form your analysis needs.
  • mutate() derives new columns from existing ones without altering a single original value. Think of it as adding new knowledge to your data, not replacing old knowledge.

mutate() — Creating a New Column

  • mutate() appends the new column to the right of the tibble. Every existing column is preserved exactly as it was.
# here we decode the integer sex code into a human-readable label.
ideal |>
  mutate(sex_label = if_else(CalfSex == 1, "Male", "Female")) |>
  select(CalfID, CalfSex, sex_label, Weight, ReasonsLoss1) |>

mutate() — Multiple Columns at Once

  • A single mutate() call creates as many columns as you need. Later expressions in the same call can reference columns created earlier, so the order of your derivations matters and can build on itself.
ideal |>
  mutate(
    sex_label  = if_else(CalfSex == 1, "Male", "Female"),
    weight_lbs = round(Weight * 2.205, 1),      # convert kg to pounds

mutate() with case_when()

  • case_when() is R’s vectorised multi-condition classifier — the equivalent of Excel’s nested IFS.
  • Each condition is evaluated top-to-bottom; the first TRUE match determines the value assigned to that row.
ideal |>
  dplyr::mutate(
    anaemia_class = case_when(
      is.na(ManualPCV) ~ "Unknown",  # always test for NA first
      ManualPCV < 24 ~ "Severe anaemia",
      ManualPCV < 30 ~ "Mild anaemia",
      TRUE  ~ "Normal" # the catch-all
    )
  ) |>

case_when() — The Mental Model

Each line is one rule: **“When** this condition is true → assign this value.”

mutate(
  new_column = case_when(
    condition_1  ~ "value when condition_1 is TRUE",
    condition_2  ~ "value when condition_2 is TRUE",
    condition_3  ~ "value when condition_3 is TRUE",
    TRUE         ~ "value for everything else"
  )
)

Rules are tested in order. The moment a condition matches, R stops testing — so place the most specific conditions first, the broadest last.

group_by() & summarise()

Most research questions are comparisons: does this outcome differ across groups? group_by() and summarise() are how tidyverse answers them

summarise() — The Baseline

Without a grouping, summarise() collapses the entire dataset to a single summary row. This is your reference point.

ideal |>
  summarise(
    n_visits      = n(),
    n_calves      = n_distinct(CalfID),
    mean_weight   = round(mean(Weight,    na.rm = TRUE), 1),
    mean_pcv      = round(mean(ManualPCV, na.rm = TRUE), 1),
    theileria_pos = sum(`Theileria.spp.` == 1, na.rm = TRUE)
  )
# A tibble: 1 × 5
  n_visits n_calves mean_weight mean_pcv theileria_pos
     <int>    <int>       <dbl>    <dbl>         <int>
1      270       30        39.3     30.8           148

na.rm = TRUE tells R to ignore missing values when computing the mean. Without it, any column containing a single NA returns NA as the summary.

group_by() + summarise()

  • Add group_by() and every summary is computed independently for each group. The result is one row per unique combination of grouping variables.

  • Call ungroup() afterwards to release the grouping structure.

result <- ideal |>
  group_by(sublocation, ReasonsLoss1) |>
  summarise(
    n_calves    = n_distinct(CalfID),
    mean_weight = round(mean(Weight,    na.rm = TRUE), 1),
    mean_pcv    = round(mean(ManualPCV, na.rm = TRUE), 1)
  ) |>
  ungroup() |>
  arrange(sublocation, ReasonsLoss1)

Common Summary Functions

Function Computes
n() Number of rows in the group
n_distinct(x) Number of unique values
sum(x, na.rm = TRUE) Total — events, counts, doses
mean(x, na.rm = TRUE) Average
median(x, na.rm = TRUE) Median — robust to outliers
sd(x, na.rm = TRUE) Standard deviation
min(x) / max(x) Minimum and maximum
sum(is.na(x)) Count of missing values
mean(condition) Proportion satisfying a condition

Every numeric summary on real data should carry na.rm = TRUE. Missing values will otherwise silently propagate as NA in your output.

Joins

Real datasets live across multiple tables. Joins are how you reunite them — matching rows from two sources on a shared key, without manually copying values between files.

Why Joins?

The IDEAL dataset records one row per calf visit. A separate table holds sub-location metadata — division, altitude, administrative boundaries. Joining enriches every visit row without duplicating the metadata by hand.

The Four Join Types

Join Result
left_join(x, y) All rows from x; matched columns from y; unmatched rows get NA
right_join(x, y) All rows from y; matched columns from x
inner_join(x, y) Only rows with a match in both tables
full_join(x, y) All rows from both; unmatched cells get NA

left_join() is the workhorse of data analysis. It preserves your primary dataset intact and attaches reference information wherever a match exists. Unmatched rows are retained with NA, not silently lost.

left_join() — Attach Without Losing Rows

# summarise to sublocation level, then attach geographic metadata
calf_summary <- ideal |>
  group_by(sublocation) |>
  summarise(
    n_calves = n_distinct(CalfID),
    mean_pcv = round(mean(ManualPCV, na.rm = TRUE), 1)
  ) |>
  ungroup()

calf_join <- calf_summary |>
  left_join(sublocation_info, by = "sublocation")

Yiro West has no entry in sublocation_info, so it receives NA for division and altitude_m. It is still present in the result — left_join() never drops a row from the left table.

anti_join() returns only the rows that failed to match. It is a data quality diagnostic. Run it before any analysis that depends on a join to catch missing reference records.
